Motion Matching & Motion Graphs on the Unitree G1

A testbed comparing two classic data-driven controllers — motion matching (MM) and motion graphs (MG) — driving the G1 humanoid on LAFAN1 mocap, with a phase-segmented, world-anchored jump skill. Below: the source data, two path experiments for each method, and the method details.

All motion is kinematic playback of retargeted mocap fragments (no physics). Blue dots = path corners; orange box = obstacle to jump over.

1 · Source data

Two G1-retargeted LAFAN1 sources (30 FPS). Every clip's first/last 45 frames (a T-pose that blends into the motion) are dropped so the T-pose never appears.

walk — walk1_subject5 (CAMDM's main walk)

A single ~258 s walk clip → a unimodal gait. Natural arms-down posture.

jump — walk_jump_walk (walk → jump → walk)

Short, nearly straight clips that walk, jump, and keep walking. Used for the jump skill.

2 · Experiment 1 — follow a square path

Path (0,0)→(6,0)→(6,6)→(0,6)→(0,0)→(6,0) (a 6×6 m loop, last edge re-traced).

Motion graph beam-planned corners

Each corner is reached by a beam-search plan (plan_to); the path closes cleanly (bbox 6×6). Corners are rounded because the single walk clip has no sharp turns (~5 m turn radius).

Motion matching reactive go-to-point

No planner: the nearest-neighbour query simply aims at the current corner. MM's trajectory features make it steer well, so it also traces the 6×6 square (slightly wobblier).

3 · Experiment 2 — same path, jump over a box at (3,0)

A box sits at (3,0), which the path crosses twice (the two →(6,0) legs), so the robot must jump over the same box twice.

Motion graph beam-planned approach to each jump

Both jumps land on the box (apex within ~0.2 m): each jump's pre-take-off entry is reached by a beam-search plan, so the apex is anchored to the box even after the loop.

Motion matching reactive jump trigger

1st jump lands (~0.2 m) but the 2nd misses by ~0.8 m: with no planner, MM cannot precisely return to the box after the loop. This is the core MM↔MG difference.

Why MG can and MM cannot guarantee the second jump. Reaching a fixed world point with a specific facing after a loop needs look-ahead. MG has a beam-search planner over its graph; MM is purely reactive (one nearest-neighbour per step). A beam search over MM's feature database would give MM the same guarantee — it just isn't built here.

4 · Method

Pipeline

Both controllers stitch mocap fragments under a shared backbone: a planar root alignment keeps the world path continuous at every cut; joint angles + root orientation are cross-faded over 12 frames; a post-process smooths the root (Savitzky–Golay on x,y) and runs foot-lock IK to remove sliding. Forward kinematics (MuJoCo) gives foot positions for features and the offline renderer.

Motion matching (MM)

  • Features (27-D, root-local): future root offset + facing at horizons {10,20,30} frames (trajectory, 12-D) and both-feet positions/velocities + root velocity (pose, 15-D); z-scored with per-group weights.
  • Criterion: query = [trajectory-from-command | pose-of-current-frame]; pick the nearest neighbour in the weighted feature space (cKDTree), with hysteresis (only switch if clearly better than continuing). Reactive — no multi-step look-ahead.

Motion graph (MG)

  • Edges: a per-frame pose+velocity descriptor (PCA-reduced to 16-D) connects frames whose descriptors are close (good blend points). Jump frames are excluded as normal targets, so locomotion never accidentally jumps.
  • Greedy following: every few frames pick {continue} ∪ {transitions} by cost(f)=‖v_local(f) − desired_local‖ + penalty (1 candidate, reactive).
  • Beam-search planning (plan_to): the only multi-step optimizer — used for the corners and the jump approaches (see below).

Beam search — and how it differs from A*

We plan a sequence of graph moves that reaches a target (position, facing, pose) at a fixed time.

A* / DijkstraBeam search (ours)
frontiera priority queue of all open nodesonly the best B nodes per round (here B=64)
expansionpop the single most promising nodeexpand every kept node, then re-prune to B
guaranteeoptimal (with an admissible heuristic)not optimal — may prune the best path
memorycan blow up (keeps everything)bounded (B × depth)
fits when…shortest path in a tractable graphhuge state space, fixed horizon, soft costs

Our state = (clip frame, world placement, elapsed time) — effectively infinite (every frame × every world pose × time), and the cost is not just path length but also command/velocity tracking. A full A* priority queue is infeasible there, so we keep a beam of the 64 best partial plans each macro-step, scored by cost-so-far + admissible distance-to-go, and at the deadline score the goal (‖xy−target‖ + |Δyaw| + pose distance), then backtrack the best plan. So: A* is exhaustive best-first shortest-path; beam search is a width-limited, fixed-horizon trajectory optimizer. (Beam search with B=∞ and a goal test would reduce to a breadth-first / A*-like search.)

Jump skill — five phases

Flight = both feet airborne; we carve five phases around it. A jump can be entered only in ready and exited only after after (never mid-air), which also prevents accidental jumps during walking.

take-off
flight
touch-down
after (exit)

Window lengths (frames): ready 12, take-off 10, flight ≈ data, touch-down 6, after 18.

Heuristics

  • Box size: per jump, a box centred under the apex; height = foot clearance over its footprint × 0.92 (so the feet clear it). Half-extents (0.13, 0.28, ≤0.12) m — the data are low hops, so boxes are low.
  • Jump anchoring: apex ≈ entry + (entry→apex displacement). To land the apex on a box, plan the ready entry to box − displacement (corrected in both x and y).
  • Foot-lock: pin the two lowest sole spheres during steady stance via damped-least-squares IK on the 6 leg joints (root untouched). Root de-jitter: Savitzky–Golay on root x,y only (z kept so jump peaks survive).
  • Turning: single walk clip → ~5 m turn radius, so square corners are rounded (no sharp turns in the data).
