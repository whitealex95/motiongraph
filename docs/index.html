Motion Matching & Motion Graphs on the Unitree G1

A testbed comparing two classic data-driven controllers — motion matching (MM) and motion graphs (MG) — driving the G1 humanoid on LAFAN1 mocap, with a phase-segmented, world-anchored jump skill. Below: the source data, two path experiments for each method, and the method details.

All motion is kinematic playback of retargeted mocap fragments (no physics). Blue dots = path corners; orange box = obstacle to jump over.

4 · Experiment 2 — same path, jump over a box at (3,0)

A box sits at (3,0), which the path crosses twice (the two →(6,0) legs), so the robot must jump over the same box twice.

Motion graph A*-planned approach to each jump

Both jumps land on the box (apex within ~0.15 m): each jump's pre-take-off entry is reached by an A* plan, so the apex is anchored to the box even after the loop.

Motion matching A*-planned approach to each jump

Both jumps land on the box (apex within ~0.15 m). MM runs the same A* planner over its feature-NN transitions, so — like MG — it precisely re-acquires the box after the loop.

Motion matching reactive (no planner) — for contrast

1st jump lands (~0.2 m) but the 2nd misses by ~0.8 m: one nearest-neighbour per step can't precisely return to the box after the loop.

It's the planner, not the controller. Reaching a fixed world point with a specific facing after a loop needs look-ahead. Reactive MM (one nearest-neighbour per step) drifts on the second jump; giving MM the same A* planner — searching over its feature-nearest-neighbour transitions instead of MG's graph edges — lands both jumps, exactly like MG. So the double-jump guarantee comes from planning, and both a motion graph and a motion-matching feature database can be planned over identically.

5 · Method

Pipeline

Both controllers stitch mocap fragments under a shared backbone: a planar root alignment keeps the world path continuous at every cut; joint angles + root orientation are cross-faded over 12 frames; a post-process smooths the root (Savitzky–Golay on x,y) and runs foot-lock IK to remove sliding. Forward kinematics (MuJoCo) gives foot positions for features and the offline renderer.

Motion matching (MM)

  • Features (27-D, root-local): future root offset + facing at horizons {10,20,30} frames (trajectory, 12-D) and both-feet positions/velocities + root velocity (pose, 15-D); z-scored with per-group weights.
  • Criterion: query = [trajectory-from-command | pose-of-current-frame]; pick the nearest neighbour in the weighted feature space (cKDTree), with hysteresis (only switch if clearly better than continuing). Reactive — no multi-step look-ahead.
  • A* planning (plan_to): the same shared planner MG uses — but MM has no precomputed edges, so its transitions are the feature-nearest-neighbours of the current frame (the same set it would jump to at runtime), computed on the fly from the KD-tree. This is what lets reactive MM also land the second box jump.

Motion graph (MG)

  • Edges: a transition connects frames whose pose descriptors are close (good blend points). We build edges on the same 15-D pose feature MM matches on (feet positions/velocities + root velocity) — so MG and MM now share one pose space. Jump frames are excluded as normal targets, so locomotion never accidentally jumps. (Trade-off: this feet-based feature ignores arm/torso joint angles, so edges have ~35% larger upper-body discontinuity than the alternative full-joint descriptor — the cross-fade absorbs it. Switch back via config.MG_DESCRIPTOR = "joint_pca".)
  • Greedy following: every few frames pick {continue} ∪ {transitions} by cost(f)=‖v_local(f) − desired_local‖ + penalty (1 candidate, reactive).
  • A* planning (plan_to): the only multi-step optimizer — used for the corners and the jump approaches (see below).

Descriptor ablation — what actually reduces the cut "pop"?

Stress test (forced transition every 8 frames, short cross-fade) of three descriptors, close-up. The number that matters is the joint discontinuity at a cut (L2, rad), split into legs vs upper body:

Adding velocity & height helps the realized motion. mm_pose_vh appends root height and yaw-rate to the feet feature. It barely changes the average edge, but the cuts greedy actually takes drop from 0.109 → 0.099 rad — nearly matching the full-body descriptor (0.093) — because turn-rate continuity stops it from splicing a left turn into a right one. It still can't fix arms it never sees, so it doesn't reach joint_pca.

Does adding far more edges help? No — not the pop. Sweeping n_neighbors 16→120 (and a looser radius) on mm_pose grows the graph 43k→460k edges. The best cut available per node improves (0.40→0.33, more choice), but the average edge gets worse (0.58→0.71, looser cuts admitted) and the cuts greedy actually takes stay flat (~0.09–0.12) — because the controller selects by velocity/goal, not by upper-body continuity. So more edges buys routing flexibility, not smoother blends; what fixes the pop is a better descriptor (or selecting edges for continuity), not more of them.

A* planning — and how it differs from beam search

We plan a least-cost sequence of graph moves that arrives at a target (position, facing, pose). The graph node is (clip frame, world placement); an edge is a macro-step (replay K=10 frames along a continuation or transition edge, re-aligned to the world). A* runs best-first over a priority queue ordered by f = g + h:

  • g (cost-so-far) = Σ ( ‖segment velocity − go-to-target velocity‖ + transition penalty ) — wandering is expensive, progress toward the target is cheap;
  • h (heuristic) = w_pos · ‖xy − target‖ — a goal-distance estimate that pulls the frontier toward the target. A node within reach of the target is a goal; its g then absorbs w_pos·‖xy−target‖ + w_yaw·|Δyaw| + w_pose·pose-distance, and the first goal popped is returned.

A discretized closed set (frame + rounded world pose) collapses revisits so the effectively-infinite state space stays finite. The winning node chain is then replayed with cross-fades and eased onto the exact terminal pose.

Why A* needs the heuristic here: with a zero heuristic, A* degenerates to Dijkstra and, given the graph's huge branching (~28 edges/node), exhausts its budget while still only halfway to the goal. The ‖xy−target‖ heuristic is what makes it dive toward the target and arrive in a few hundredths of a second. Beam search sidesteps the missing-heuristic problem by advancing a fixed-width front in lockstep — robust, but it can prune the optimal plan and has no goal pull. We switched the planner from beam search to A*.

Jump skill — five phases

Flight = both feet airborne; we carve five phases around it. A jump can be entered only in ready and exited only after after (never mid-air), which also prevents accidental jumps during walking.

take-off
flight
touch-down
after (exit)

Window lengths (frames): ready 12, take-off 10, flight ≈ data, touch-down 6, after 18.

Heuristics

  • Box size: per jump, a box centred under the apex; height = foot clearance over its footprint × 0.92 (so the feet clear it). Half-extents (0.13, 0.28, ≤0.12) m — the data are low hops, so boxes are low.
  • Jump anchoring: apex ≈ entry + (entry→apex displacement). To land the apex on a box, plan the ready entry to box − displacement (corrected in both x and y).
  • Foot-lock: pin the two lowest sole spheres during steady stance via damped-least-squares IK on the 6 leg joints (root untouched). Root de-jitter: Savitzky–Golay on root x,y only (z kept so jump peaks survive).
  • Turning: single walk clip → ~5 m turn radius, so square corners are rounded (no sharp turns in the data).
